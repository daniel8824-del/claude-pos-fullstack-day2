4실습 Phase 1 | POS 데이터 로더

POS 데이터 명세 — SAP Integrated Business Planning (IBP) 표준 주요 지표 기반:

컬럼 (주요 지표 ID)의미예시
BASE_WEEK기술적 주 (YYYYWW)202618
PRODUCT_ID제품 SKUAMP-TONER-001
LOCATION_ID장소 (점포·DC)STORE_GANGNAM
CUSTOMER_ID고객 (백화점·마트·면세점·온라인)CUST_DEPT_LOTTE
CUSTSTORESLSQTY고객 점포 판매 수량 (판촉 포함)720
CUSTSTORESLSPROMQTY고객 점포 판촉 판매 수량 (정기 외)120
CUSTSTORESTOCKQTY고객 점포 재고 수량1500
CUSTDCSLSQTY고객 물류센터 DC 판매 수량700
CUSTDCSTOCKQTY고객 물류센터 DC 재고 수량8000

건수: 205건 (정상 200 + 이상 5). 5주 분량 (202618 ~ 202622) · 화장품 6 SKU · 8개 점포 · 4개 고객사. UTF-8 BOM 인코딩 (Excel 한글 호환).

이상 5건 — Phase 2 분석에서 발견해야 할 케이스:

  1. STOCK 음수 — 입고 누락 또는 마이너스 정합 실패 (CUSTSTORESTOCKQTY < 0)
  2. SLS=0인데 PROM>0 — 정기 판매 0인데 판촉 판매만 발생 (불가능 패턴)
  3. 재고 결품 직전 — SLS > STOCK (판매가 재고 초과)
  4. DC 판매 0인데 점포 판매 큼 — 배송 누락 또는 DC 우회 의심
  5. PROM > SLS — 판촉 수량이 전체 수량보다 큼 (정합 위반)

출처 | SAP IBP 고객 점포·DC 데이터 주요 지표 (SAP Help). POS·CDP·SCM 도메인 표준 키 지표.

회사 데이터 주의: 실 매장 매출·고객 정보는 절대 GitHub public Pages 올리지 않기. 가상 데이터(오늘 실습용 pos_data.csv)만 안전. 회사 운영 분석은 사내 git private + 사내 호스팅으로.

아래는 강사 시연 (가상 데이터 사용 시):

  1. Git 초기화 + .gitignore
    cd 실습폴더 && git init -b main
# .gitignore는 AI에게 요청
  2. 첫 커밋 (강사: /smart-commit · 수강생: 인라인)
    git add . && git commit -m "init"
  3. 신규 repo 생성 + 푸시
    gh repo create pos-analyzer-day2 --public --source . --push
  4. Pages 활성화
    gh api -X POST repos/{user}/pos-analyzer-day2/pages \
  -f 'source[branch]=main'
  5. 3분 후 라이브 URL 확보
    https://{user}.github.io/pos-analyzer-day2/ — 강의 끝나면 강사에게 공유
강사 사고 회수 — auto-accept 무한루프 $50/일: 종료 조건 없이 auto-accept 모드 + 자리 비움 → 8시간 같은 작업 변형 반복. 룰: 종료 조건 명시 + 시간 제한 + 비용 알람. 베테랑 청중도 같은 사고 가능 — 본 강의에서 정직 회수.
§ AI-Ops · 회사 추가 요청 반영 (강의안 부록)

전통 POS → AI-Ops 전환 5단계 로드맵

오늘 배운 도구를 POS 운영·개발 조직에 단계적으로 적용하면 어떤 변화가 가능한지. 본 섹션은 회사 측(시바·아모레퍼시픽) 추가 요청 반영 — 강의안 본 흐름 외 부록.

단계전통 POS (As-Is)AI-Ops (To-Be)
1매출 보고서 점장별 양식 → 본사 통합 1~2일CLAUDE.md/AGENTS.md에 통일 양식 → AI 자동 정리
2분석 요청 시 SQL 매번 새로 → 2~3일SKILL로 자주 쓰는 분석 패턴 누적 → 30분
3POS 단말 오류 → 새벽 호출 → 출동HOOK으로 자동 진단 · 자동 티켓 발행
4외부 데이터(상권·날씨·트렌드) 수동 합치기MCP로 외부 시스템 직통 → AI 자동 결합
5배포·보안 점검 매번 수동/security-review + GitHub Pages 자동화

실행 과제

  1. 데이터 표준화 — 매장별 양식 통합 (CLAUDE.md/AGENTS.md 1차)
  2. 분석 자동화 — 자주 쓰는 패턴 SKILL.md 누적
  3. 운영 자동화 — HOOK 도입 (위험 명령·실수 자동 차단)
  4. 외부 연계 — MCP로 사내 DB·상권 데이터·티켓 시스템 연결
  5. 배포 파이프라인 — 보안 점검 자동화

고려 사항 (POS 특수성)

  • 결제 정확성 100% — AI 자동화는 보조 영역부터, 결제 본 함수는 사람 승인 후
  • 24/7 영업 환경 — 배포 윈도우 제한, 점진적 롤아웃 필수
  • 매장 네트워크 분산 — 단말 다양성·오프라인 대응 고려
  • PG·카드사 연동 — 외부 의존성 변경 시 다중 검수
  • PCI-DSS 등 보안 규제 — 민감 정보 격리·로그 보존

Claude Code/Copilot HOOK의 동작 범위 | 개발자 PC 내 동작. POS 운영 시스템 자체 자동화는 MCP + 외부 자동화 도구(n8n · Airflow 등) 결합으로 단계적 진입. 오늘 배운 것 = 개발 단계 가속, 운영 시스템 자동화는 별도 트랙.
